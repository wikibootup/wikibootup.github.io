Chapter 2.1¶

데이터가 무엇인지, 왜 데이터를 묶어서 사용하는지, 그리고 데이터 인터페이스를 어떻게 구성하는지 생각해보고
연습해보는 것에 초점을 맞춰보도록 하겠습니다. 그런 관점에서 본다면 왜 데이터를 정의하기 위해서 list나
dictionary 같은 편리한 데이터 구조가 아닌 pair를 사용하는지 이해할 수 있을 것입니다.

  • pair는 최소 단위의 묶음 데이터compound data이자 어떠한 데이터 구조라도 만들 수 있는(universal)
    빌딩 블록building block으로 기능합니다.

데이터 요약
데이터 구조를 생성하는 방식contruction과 골라내는 방식selection을 나누어 생각하는 방법입니다. ( 책 2장
서문 참조 )

  • 그런 의미에서 cons는 constructor이고 car, cdr은 selector입니다.

요약의 경계Abstraction Barrier
User-defined_data_types을 떠올리게 합니다. '왜 int, char 같은 익숙한 타입에 이름을 지정해서 피곤하게
만드는 걸까'하고 생각할 지도 모르겠습니다. 하지만 문제가 복잡해질 수록, 프로그램의 구조를 수정하는 일이
빈번해질 수록, 그리고 시간이 지날 수록(이게 무슨 데이터인지 기억에서 사라질 정도로) 그것이 필요할 것입니다.

  • 물론, 데이터 타입을 예로 든 것은 예시입니다. 데이터를 요약하는 경계와는 차이가 있는 부분입니다. 요약의
    경계는 계층으로 나누어 데이터를 바라보고 설계한 (계층)구조입니다.
  • 책에서는 유리수를 예를 들어 설명합니다. 유리수를 '문제 영역에서 보느냐(+, - 등 연산), 개념 영역에서
    보느냐(유리수는 분자와 분모로 구성), 컴퓨터 데이터 영역(유리수는 pair로 구성)에서 보느냐' 등에 따라
    그 계층을 나눌 수 있습니다.

Church encoding을 소개하는 문제입니다. Exercise 2.4에서 본 것처럼 다른 것들도 값(숫자) 없이
함수만으로 데이터 구조를 포함한 여러 기능을 만들 수 있습니다. 그렇지만 프로세스가 수행되면서
각각의 기호(변수명)가 지금 어떤 상태인지, 그리고 어떤 결과값이 만들어지고 있는지를 생각해야 해서
쉽지가 않습니다. 마치 '젓가락 한 개로 라면 먹는 기분이 이런게 아닐까' 싶습니다. Test reference

  • Exercise 2.4에서 만든 pair 구현은 Church Pairs와 같습니다.

Church Booleans
$True \equiv \lambda a. \lambda b.\ a$,
$False \equiv \lambda a. \lambda b.\ b$
$And \equiv \lambda p. \lambda q. p\ q\ p$
$Or \equiv \lambda p. \lambda q. p\ p\ q$

위의 식은 함수로 구현한 Boolean 식들expressions입니다. True, False는 양자택일의 상황에서
하나를 선택한 것일 뿐입니다. 하지만 And와 Or는 다소 와닿지 않을 수 있습니다. 여기서 생각해두면
좋은 것이 And와 Or 식은 인자로 술어의 판단 결과, 즉 True 또는 False만 받는다는 것입니다. 이것을
생각해보면 And의 인자에는 False가 단 하나만 포함되어 있어도 False이고( And 구현 중
(p (q p)) 부분), Or의 경우에는 False 인자만 있는 경우에만 False가 됨을( Or 구현 중
(p (p q)) 부분) 이해할 수 있습니다.

  • 언제부턴가 너무나도 당연하게 1을 true, 0을 false로 판단해서 그렇지 프로그래밍을 처음 배우는
    사람들에게는 이 개념 역시 낯설지 모릅니다.

Church Numerals
$0 = \lambda f. \lambda x.\ x$,
$n = \lambda f. \lambda x.\ f^n x$,
$Succ \equiv \lambda n. \lambda f. \lambda x. f\ (n\ f)\ x$,

$value \equiv \lambda v. (\lambda h. h\ v)$
$inc \equiv \lambda g. \lambda h. h\ (g\ f)$
$const \equiv \lambda u. x$
$Identity \equiv \lambda u.u$
$Pred \equiv \lambda n. \lambda f. \lambda x. n\ inc\ const\ identity$

$Plus \equiv \lambda m. \lambda n. \lambda f. \lambda x.m\ f\ (n\ f\ x)$,
$Minus \equiv \lambda m. \lambda n. (n\ Pred)\ m$,

  • mult, exp 계산은 수학에서의 지수법칙이 잘 적용됩니다.
    ( 위키백과 Church encoding 참조 )
  • 위의 Pred는 모듈($inc,\ const,\ identity$)로 축약해놓은 형태입니다. 실제로 저렇게 모듈로
    pred 함수를 구현하려면 변수 범위scope에 주의해야 합니다. 함수 이름(inc 등)보다는 각
    함수끼리 어떻게 상호작용해서 (n-1)의 구현이 가능한지 보는게 좋을 것 같습니다.
  • Succ은 n이라는 함수에 f를 적용합니다.
    n : church 숫자(함수식) f : 디코딩 시에 적용할 함수
    x : 디코딩 시에 적용될 초기값(실제 수)으로 (f (n f))를 통해 최종적으로 만든 함수에 인자로
    넣어줄 값

이것이 그냥 Church가 아니라 Church encoding인 이유는 decoding을 통해서 수로 표현할 수
있기 때문입니다. church numerals 식에서의 f가 바로 디코딩의 핵심입니다.

  • Decoding : $f = inc\ x = x + 1$
  • Encoding :
    if x = 0? then zero else ((succ (((encoding (- n 1)))) x)

Exercise 2.9¶

위의 문제에서 예로 든 간격들intervals을 다시 정의하겠습니다.

A = [lower-bound-of-A, upper-bound-of-A] = [la, ua], 
B = [lower-bound-of-B, upper-bound-of-B] = [lb, ub],
A + B = [la+lb, ua+ub],
A - B = [la-ub, ua-lb]

위의 정의를 토대로 문제를 풀어보면 구간값의 덧셈은 가산성Additivity이 성립함을, 뺄셈은 뺄셈이 아닌 반대
방향으로의(덧셈으로) 가산성Additivity 성립함을 알 수 있습니다( 뺄셈의 경우 정확히 말하면 가산성이
성립하지는 않습니다. 규칙이 있을 뿐입니다. ).

width(A+B) = $\frac{(ua+ub)-(la+lb)}{2}$ = $\frac{ua-la}{2} + \frac{ub-lb}{2}$ = width(A) + width(B)

width(A-B) = $\frac{(ua-lb)-(la-ub)}{2}$ = $\frac{(ua+ub)-(la+lb)}{2}$ = width(A+B)

width(A) * width(B) = $\frac{(ua-lb)*(la-ub)}{2}$

하지만 곱셈, 나눗셈의 경우에는 이런 규칙이 성립하지 않습니다. 먼저 구간값의 곱셈의 경우 위에서 적은
width(A) * width(B)식의 결과와 달리 width(A*B)는 mul-interval연산으로 인해 하한과
상한의 부호sign에 따라 9가지 경우의 수가 발생하여 결과가 달라지기 때문입니다( 즉, 규칙이 성립하지 않음 ).
또한 책의 구현에서 나눗셈은 곱셈을 내부 프로시저로 가지고 있으므로 나눗셈 역시 규칙을 가지지 않습니다.

'구간에 0이 들어있는가'가 뜻하는 것은( an interval that spans zero ) 하한과 상한 사이에 0이 포함되어
있는지를 물어보는 것입니다. 즉, 하한과 상한은 값이 0일 수도 있지만 부호가 다를 수도 있다는 것입니다.
그리고 0이 포함되면 안되는 이유는 수를 0으로 나눌 수 없다는 이유도 있지만 이 문제에서 정의한 구간 나눗셈
연산에서는 또 다른 문제가 있습니다.

하한과 상한의 부호가 서로 다를 경우 나눗셈 연산에서 하한과 상한이 뒤바뀌는 현상이
발생하기 때문입니다.

Exercise 2.13¶

문제에서 제시한 대로 두 구간의 성분이 모두 '양수'라고 가정하면 두 구간의 각 성분간의 곱들의 대소비교가
되지 않은 모호한 상태를 피할 수 있습니다. 이 때의 조건은 Exercise 2.11에서 구한 mul-interval
프로시저의 첫번째 조건 (mul-interval-9-cases pp pp)과 같습니다. ( Exercise 2.11 문제
풀이 참조 )

Center_of_interval_of_a = ca = (lower_interval_of_a + upper_interval_of_a) / 2

Tolerance_of_a = ta =
(Percent_tolerance_of_a / 100.0) * Center_of_interval_of_a = (pa / 100.0) * ca =

X = [cx - tx, cx + tx]
Y = [cy - ty, cy + ty]
XY = [(cx - tx)(cy - ty), (cx + tx)(cy + ty)]

하한값을 계산해보면,
$(cx - tx)(cy - ty) = $
$cx(1 - \frac{px}{100.0}) * cy(1 - \frac{py}{100.0}) = $
$(cx * cy)(1 - \frac{py}{100.0} - \frac{px}{100.0} + (\frac{px}{100.0})(\frac{py}{100.0})) = $
$(cx * cy)(1 - \frac{px + py}{100.0} + \frac{px * py}{10000.0})$

여기서 px, py(허용 오차)가 매우 작다고 가정하면 $\frac{px * py}{10000.0} \approx$ 0

상한의 경우에도 같은 방식으로 계산하면 X와 Y의 곱의 결과는 아래 구간과 근사값을 이룹니다.
XY = [ (cx * cy)(1 - (px + py / 100.0)), (cx * cy)(1 + (px + py / 100.0)) ]

Exercise 2.14, Exercise 2.15, Exercise 2.16¶

구간연산은 완벽히 정확한 값을 계산하기 어려운 컴퓨터의 문제점( Observational error )의 대안으로 쓰일

결과는 다른 경우가 생깁니다. 그 예를 보여주는 것이 Exercise 2.14입니다.

오차범위가 0에 수렴하기는 커녕 꽤 큰 범위를 가지고 있음을 알 수 있습니다. 이것은 처음부터 정확한 값이
아닌 구간들로 연산을 수행함에 따라 오는 일종의 부작용일 것입니다. 따라서 수학적으로는 같은 식이라해도
구간 사이에 곱/나눗셈/제곱 등의 연산은 그 과정에서 서로 다른 오차범위를 가지게 된다고 '생각'합니다.

위키백과에 나온 예를 들면 수학적으로는 $x^2+x = (x+\frac{1}{2})^2-\frac{1}{4}$이지만 구간연산을 하면 둘의 결과는
차이가 생깁니다. 또한 구간연산에 있어서는 등호 오른쪽의 식이 더 좋은 식입니다. 그 이유는 왼쪽 식보다
오차범위가 적게 나오기 때문입니다. 왜그런가하면 구간연산의 제곱과 곱으로 인해 오차범위가 넓어지기
때문입니다. 다르게 말하면 상대적으로 구간연산 수가 적은 우측 식이 오차범위를 좁힐 수 있게됩니다.

  • $x^2+x$의 연산과정 중에서 하한이 0인 이유는 '실수의 짝수승의 최소값이 0이기 때문입니다.'

;; it is not a iterative implementations actually because the process have
;; a position to return, in other words process makes stacks to return,
;; that is, it is rather a recursive, not iterative.
(define (deep-reverse-5 items)
  (define (iter items result)
    (if (null? items)
        result
        (if (pair? (car items))
            (let ((x (iter (car items) '())))
              (iter (cdr items) (cons x result)))
            (iter (cdr items) (cons (car items) result)))))

  • 내부에 iter의 네이밍은 적절하지 않다고 생각합니다. 왜냐하면 이 프로시저를 돌리면 프로세스는 돌아가야
    할 곳이 생기기 때문입니다. 다르게 말하면 프로세스는 stack 공간을 만들면서 자라나므로 재귀적 구현이라고
    보는게 적절하다고 판단합니다. * 하지만 다시 볼 때 다시 생각할 수 있도록 남겨두겠습니다.

여기서 iter의 인자 값들의 흐름은 아래와 같습니다.

items (((0 1) 2) (3 4)) ((0 1) 2) (0 1) (1) () (2) () ((3 4)) (3 4) (4) () ()

result () () () (0) (1 0) ((1 0)) (2 (1 0)) ((2 (1 0))) () (3) (4 3) ((4 3) (2 (1 0)))

과정 1.¶

재귀적으로 iter이 호출되다가 let 문장에서 (iter (0 1) nil)이 호출되면, 함수 내부 연산 중
(car items) = (car (0)) = 0 $\neq$ pair이므로 술어를 만족하지 않습니다. 그래서 아래 연산을
거칩니다.

items <- (0 1)

x <-(iter (cdr items) (cons (car items) result))))) = (iter (1) (cons 0 '())) = (1 0)

과정 2.¶

이 결과는 (iter ((0 1) 2) '())의 문제 안으로 반환됩니다. 따라서 let으로 이어지는
식expressions은 아래처럼 변합니다.

(let ((x (1 0))) (iter 2 (cons x '())))

과정 3.¶

이런 과정('과정1,2')을 쭉 밟다보면 처음 함수가 분리된 위치까지 올라가게 됩니다. 그 다음은 더 이상의
데이터(cdr items)가 없으므로(=nil) result를 반환하며 프로세스는 종료됩니다.

(define (fringe-4 T)

하지만 weight 대신 structure로 명명한 이유는 branch 아래에 또다른 모빌이 있을 수도 있기

a.¶

다음은 고르개selector입니다.

In [17]:
(define (left-branch m)
  (car m))

;; A list has null to the last index. therefore if one element is left,
;; it will returns a pair.
(define (right-branch m)
  (cadr m))

(define (branch-length b)
  (car b))

;; if a branch has a mobile as structure
;; 'cdr' returns 2 level-enclosed lists. (( m ))
;; and if it has just a weight,
;; 'cdr' returns a value that is not enclosed.
(define (branch-structure b)
  (let ((structure (cdr b)))
    (if (not (pair? structure))
        structure
        (car structure))))

branch-structure의 경우 '추weight를 가지냐, 모빌을 가지냐'에 따라 데이터를 다르게
뽑아내야 하므로 조건이 붙습니다.

b.¶

구현한 데이터 요약(constructros, selectors)을 이용해서 모빌의 전체 무게를 구해보기로 합니다. 모빌은 맨
마지막 branch에만 '추'가 있으므로 이 마지막 branch의 무게를 모두 더하기만 하면 됩니다. '마지막 branch인지,
아닌지'를 판단하기위해 '그것이 아래로 모빌을 가지고 있는지'의 유무를 확인할 수 있는 술어들을 만들어 보겠습니다.

In [1]:
;; does-a-structure-of-a-branch-contain-mobile-or-not?
(define (structure-mobile? structure)
  (pair? structure))

;; does a branch contain mobile?
(define (contain-mobile? b)
  (structure-mobile? (branch-structure b)))

;; does not a branch have a mobile?
(define (last-branch? b)
  (not (contain-mobile? b)))

    부가적인 요소(관사, 전치사, 접속사)와 생략가능한 요소(A or not? 대신 A?)를 생략했습니다. 그리고
    언어(여기선 scheme)의 관습적인 요소( 의문문 표현의 is대신 ?)를 적용했습니다. * 이런 것들은 '암묵적 또는
    명시적 합의'를 통해서 이루어질 것입니다.

모빌의 무게를 구하려면 branch의 무게를 알아야 합니다. 하지만 branch가 아래로 모빌을 가진다면 다시
모빌의 무게 역시 알 수 있아야 합니다. 따라서 모빌의 무게를 구하는 프로시저와 branch의 무게를 구하는
